# FLUX — AI ネイティブ計算基盤

**FLUX** は、AIシステム（LLM）がFTL（FLUX Text Language）で計算グラフを生成し、それを形式的に検証して最適なマシンコードにコンパイルする実行アーキテクチャです。

**LLMがFTLテキストを生成。システムがバイナリにコンパイル。形式的に検証済み。最適。**

## 設計公理

```
1. コンパイル時間は無関係           → 網羅的検証、超最適化
2. 人間の可読性は無関係             → LLMはFTL（構造化テキスト）で作業、
                                      システムがバイナリにコンパイル
3. 人間の補償措置は不要             → デバッグなし、例外処理なし、
                                      防御的プログラミングなし
4. コード生成のパフォーマンスは二次的 → LLMの反復回数無制限、
                                      分析の深さ無制限
5. 創造性が望まれる                 → AIは新しい解を発明すべき、
                                      既知のパターンを再現するだけではない
6. 検証における実用主義             → タイムアウト付き段階的証明戦略、
                                      決定不能 → エスカレーション、無限ループではない
```

## アーキテクチャ

```
要件（自然言語、スコープ外）
    │
LLM（プログラマー — 人間を置き換える）
    │  FTL（FLUX Text Language） — 構造化テキスト
    ▼
FLUXシステム
    ├─ FTLコンパイラ（テキスト → バイナリ + BLAKE3ハッシュ）
    ├─ バリデータ（構造 + 型 + エフェクト + リージョン）
    │    失敗 → LLMへのJSONフィードバック（提案付き）
    ├─ 契約証明器（段階的: Z3 60秒 → BMC 5分 → Lean）
    │    反証 → LLMへの反例
    │    決定不能 → LLMへのヒントまたはインキュベーション
    ├─ プール / 進化（発明/発見用）
    │    LLMへのフィットネスフィードバック（相対メトリクス）
    ├─ 超最適化器（3段階: LLVM + MLIR + STOKE）
    │    ホットパス最適、残りはLLVM -O3品質
    └─ MLIR → LLVM → ネイティブマシンコード
    │
┌───┴────┬──────────┬──────────┐
ARM64   x86-64    RISC-V     WASM
```

## ノードタイプ

| ノード | 機能 |
|--------|------|
| **C-Node** | 純粋計算（ADD, MUL, CONST, ...） |
| **E-Node** | 副作用、正確に2つの出力（成功 + 失敗） |
| **K-Node** | 制御フロー: Seq, Par, Branch, Loop |
| **V-Node** | 契約（SMT-LIB2） — コンパイルには証明が必須 |
| **T-Node** | 型: Integer, Float, Struct, Array, Variant, Fn, Opaque |
| **M-Node** | メモリ操作（リージョンに紐付き） |
| **R-Node** | メモリライフタイム（アリーナ） |


## 中核原則

**プログラマーとしてのLLM:** LLMが人間のプログラマーを置き換えます。FTLテキストを提供します（バイナリやハッシュではなく）。システムがFTLをバイナリグラフにコンパイルし、BLAKE3ハッシュを計算し、JSONフィードバックを返します。

**完全な正確性:** コンパイルされたすべてのバイナリは形式的に検証されます。ランタイムチェックはゼロ。契約は段階的証明戦略（Z3 → BMC → Lean）で証明されます。

**探索的合成:** AIは1つではなく、数百のグラフを生成します。正確性がフィルター、創造性がジェネレーター。遺伝的アルゴリズム（GA）が主要なイノベーションエンジンで、LLMが初期化と的を絞った修復を提供します。

**超最適化:** 3段階（LLVM -O3 → MLIRレベル → STOKE）。ホットパスは手書きアセンブリより優秀。現実的: 純粋なLLVM -O3に比べて全体で5-20%の改善。

**コンテンツアドレス方式:** 変数名なし。アイデンティティ = コンテンツのBLAKE3ハッシュ（システムが計算）。同じ計算 = 同じハッシュ = 自動重複排除。

**生物学的変異モデル:** 欠陥のあるグラフはインキュベーションゾーンで隔離され、さらに開発されます。変異の上の変異が「悪い」ものを「特別な」ものに変えることがあります。最終バイナリのみが証明可能に正しい必要があります — そこへの道はエラーを通ってもよいのです。

## 要件タイプ

```
翻訳       「マージソートで並べ替え」              → 直接合成（1グラフ）
最適化     「できるだけ速く並べ替え」              → パレート選択（多数のバリアント）
発明       「sort()を改善、新しいものを発明」      → 探索的合成 + 進化
発見       「性質Xを持つ計算を見つけよ」           → グラフ空間でのオープンサーチ
```
